    Existing host access

    Enter the IP address or DNS name of the existing RHEL 9 or RHEL 10 machine where MicroShift will be installed. This is the host you SSH into, not a future cluster or API hostname.

    Local VM provisioning

    Enter the local path on the Cockpit host to a RHEL qcow2 cloud image that can be imported with virt-install --import.
    Provide a direct URL the Cockpit host can fetch with curl. This can be a signed or authenticated Red Hat image URL, or an internal mirror URL.
    Optional. If left empty, the cached qcow2 filename is derived from the download URL.
    For Red Hat Build of MicroShift 4.21, use a RHEL 9.6 cloud image for the supported install path. RHEL 10.0 is Technology Preview only, and newer RHEL 10 minors can fail because the required RPMs are not exposed through the enabled MicroShift repositories. Compatibility table
    Select the host bridge the new VM should attach to, for example the same bridge your other local VMs use for LAN access.
    Enter the static IPv4 or IPv6 address you want the new MicroShift VM to use on your network, including the prefix length. Example: 192.168.1.250/24.
    Enter the router address the new VM should use to reach systems outside its local subnet. Example: 192.168.1.1.
    Enter the DNS servers the new VM should use for name resolution, separated by commas. Example: 192.168.1.142, 192.168.1.242.
    Applies the same Stakkr-style weighted CPU share tiers used by the OpenShift plugin when provisioning the local MicroShift VM.
    Local provisioning divergence
    Creating a host locally with libvirt/KVM is a Cockpit convenience layer. The actual MicroShift install inside that guest still follows the documented Red Hat host-based RPM workflow. Downloaded images are cached on the Cockpit host under the plugin runtime directory.

    Authoritative install model
    The final install still follows the Red Hat Build of MicroShift RPM workflow. Automated registration is a convenience layer for preparing the target host before dnf install microshift openshift-clients.
    The preflight step validates the selected operating model. Existing-host deployments are checked over SSH. Create-host deployments validate local libvirt prerequisites first, then apply the same in-guest MicroShift checks after the VM is provisioned.

    Host preparation

    Preflight expectations

    • Existing-host deployments require a reachable RHEL 9 or RHEL 10 machine with non-interactive sudo.
    • Create-host deployments require local libvirt/KVM, a dir-backed storage pool, a bridge, and either a usable local RHEL cloud image or a direct downloadable qcow2 URL.
    • The final MicroShift host must either already expose the required RPM repositories or be registered through the automated package-access mode before installation starts.
    • The plugin will write /etc/crio/openshift-pull-secret and /etc/microshift/config.yaml on the final MicroShift host.

    Core MicroShift configuration

    If left empty, the target host's current hostname is used for the external kubeconfig path and serving certificates.
